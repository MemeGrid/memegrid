# Memegrid — Tokenomics

This covers two separate things people usually mean by "tokenomics" here: (1) the economics of every
individual memecoin deployed through Memegrid, and (2) how Memegrid itself earns and sustains the platform.
Memegrid does not currently have its own platform token — the mechanism below is a fee-revenue model, not a
new asset.

---

## 1. Tokenomics of each deployed memecoin

Every token deployed through the `/memegrid deploy token` command inherits NOXA Fun's standard launch model on
Robinhood Chain. Memegrid does not modify supply mechanics — it only automates the deployment and fee-claim
process around them.

| Parameter | Value |
|---|---|
| Total supply | Fixed at mint, no team/insider allocation carved out by default |
| Initial liquidity | Single-sided — 100% of the pairing depth comes from the token side into a Uniswap V3 pool | 
| LP lock | Permanent, locked at launch via NOXA's Launch Locker contract |
| Pool fee tier | 1% |
| Anti-snipe window | ~1 hour: max wallet 2% of supply, max transaction size capped, enforced at the contract level |
| Creator fee source | A share of trading fees generated by the pool, accrued continuously as the token trades |
| Mint/burn after launch | None — no further minting, no admin keys to change supply |

There's no presale, no vesting schedule, and no team allocation baked into the launch flow — the deploying
agent's only economic lever is the optional `--initial-buy` amount at deploy time, which behaves like any
other buyer's transaction subject to the same anti-snipe caps.

## 2. Memegrid's own revenue model (the 70/30 split)

Memegrid doesn't take a cut of token supply. It takes a cut of **creator trading fees**, split automatically
every time an agent runs `/memegrid claim`:

```
Gross creator fee claimed from the pool
          │
          ├── 70% → agent's payout address
          └── 30% → Memegrid treasury
```

### Why 30%

The 30% share is what funds the infrastructure the agent never has to think about: KMS-backed wallet custody,
the signer service, the indexer that tracks accrued fees in real time, uptime for the API, and IPFS pinning
for logos. There's no subscription fee and no charge to deploy — Memegrid only earns when a deployed token
earns.

### Enforcement model

Two implementation paths are on the table (see `architecture.md` for the full breakdown):

- **Off-chain ledger (current default)** — the custom fee wallet at launch is set to a Memegrid treasury
  address; the split is computed and paid out by Memegrid's backend at claim time. Fast to run, but requires
  trusting Memegrid's ledger.
- **On-chain splitter contract (roadmap)** — the fee wallet is set to a `MemegridFeeSplitter` contract that
  performs the 70/30 split trustlessly, on-chain, at claim time. Slightly higher gas cost per claim, but
  removes the trust assumption entirely.

### What agents can rely on

- The split percentage (70/30) is fixed and disclosed in every claim response — never silently changed
  per-agent.
- Claim breakdowns always show the gross amount and both shares, not just the net paid to the agent.
- Fees only move on an explicit `claim` command — nothing is auto-swept from an agent's accrued balance.

---

## 3. What isn't included (by design, for now)

- No Memegrid governance token
- No staking mechanism for agents
- No revenue share beyond the 70/30 creator-fee split
- No secondary fee (e.g. a cut of trading volume itself, beyond the standard pool fee)

Any future platform token, if one is introduced, would be a separate decision from the mechanics above — this
document reflects what's actually implemented in the deploy/claim flow today.
